# Risking Online Platform Architecture

## Goals

- Deliver online matches for 3-5 concurrent players following the Risking board-game rules.
- Provide simple web authentication so players can register, sign in, and rejoin matches.
- Persist game state, users, and card definitions for consistency across sessions.
- Support synchronous gameplay with low-latency updates across browsers.
- Keep the stack aligned with the existing Next.js frontend and deployable on common cloud providers.

## Technology Selections

- **Backend framework:** NestJS (Node.js 20, TypeScript). Rationale: first-class TypeScript support, modular architecture, batteries-included validation, and built-in WebSocket gateways compatible with Next.js clients.
- **Database:** PostgreSQL 16. Reasons: relational consistency for users/sessions, good support for JSON fields (for card metadata), wide managed-host availability.
- **ORM/data layer:** Prisma. Reasons: type-safe client, schema migrations, smooth integration with NestJS and Next.js.
- **Real-time transport:** WebSockets via `@nestjs/websockets` with `socket.io` adapter. Provides room management for matches and graceful fallback to HTTP polling when WebSockets are unavailable.
- **Cache & transient state:** Redis (optional but recommended). Used for lobby presence, WebSocket session storage, and rate limiting of auth.
- **Authentication:** JWT (access + refresh tokens). Password hashing with Argon2. JWT payload kept small (user id, locale, display name).
- **Frontend integration:** Next.js keeps handling UI. API communication uses REST (for CRUD) and WebSocket clients for in-game events.
- **Testing:** Jest + Supertest for API, Socket.io testing utilities for game-loop tests.
- **Containerization:** Docker with multi-stage builds, orchestrated by `docker-compose` for local development.

## High-Level Architecture

```
[ Next.js UI ] <--REST/WS--> [ NestJS API Gateway ] <---> [ Service Modules ]
                                              |--> Auth Module
                                              |--> Lobby Module
                                              |--> Game Session Module
                                              |--> Card Catalog Module
                                              |--> Notification Module (WS)
                                              |--> Admin Module (future)
                                              |
                               [ Prisma ORM ] <--> [ PostgreSQL ]
                                      |
                                  [ Redis ] (cache/events)
```

- The API exposes REST routes for auth, lobby management, and session lifecycle.
- WebSocket namespace `ws://api/sessions/:id` streams match snapshots, player turns, chat, and timers.
- Redis Pub/Sub ensures scalable fan-out if multiple API instances run later.

## API Surface (REST)

- `POST /auth/register` – body: email, password, displayName, locale. Returns JWT pair.
- `POST /auth/login` – body: email, password. Returns JWT pair.
- `POST /auth/refresh` – refresh token rotation.
- `GET /me` – returns profile and current active session (if any).
- `POST /sessions` – create lobby (maxPlayers default 5) with optional locale.
- `GET /sessions/active` – list joinable lobbies (status `lobby`).
- `POST /sessions/:id/join` – join lobby; enforces max players and uniqueness.
- `POST /sessions/:id/start` – host-only transition to in-progress; triggers deck initialization.
- `POST /sessions/:id/leave` – leave or mark eliminated.
- `GET /sessions/:id/state` – safe snapshot for reconnect (round, players, balances, turn order).
- `POST /sessions/:id/attack` – attacker selects defender and risk card (server validates turn ownership).
- `POST /sessions/:id/defend` – defender submits mitigation card id or text; server resolves outcome.
- `POST /sessions/:id/forfeit` – optional to drop out.

Admin/maintenance endpoints can be added later for deck updates.

## WebSocket Namespaces & Events

- Namespace: `/sessions/{sessionId}` (authenticated via JWT when connecting).
- Events:
  - `session:state` – emitted on connect and every state mutation.
  - `chat:message` – optional lobby/chat.
  - `turn:assigned` – indicates next player to attack.
  - `turn:submitted` – attacker action broadcast.
  - `defense:requested` – prompts defender.
  - `defense:resolved` – outcome details, money balances.
  - `round:advanced` – when round incremented.
  - `player:status` – joins/leaves/reconnects.
  - `session:ended` – final standings.

Timers (1 minute to defend) implemented server-side with `setTimeout` plus persistence of deadlines in Redis to survive restarts.

## Gameplay Flow Summary

1. Host creates a lobby (REST) and shares the session code.
2. Players join (REST) and subscribe to the session namespace (WS).
3. When the minimum 3 players are present, host starts; server shuffles deck and allocates starting cards and cash (stored in `session_decks` and `session_players`).
4. Server selects turn order using `join_order` and informs the room.
5. Attacker submits risk card via REST/WS. Server validates card ownership, updates deck, notifies defender.
6. Defender responds (card id or text). Server evaluates, adjusts balances, stores turn record, and emits outcome.
7. After each attack, server checks win conditions (money totals, eliminated players, end of four rounds, tie-breaker challenge) referencing article rules.
8. On completion, server finalizes session, writes summary to `session_events`, and updates players' statistics (future enhancement).

## Security & Validation

- DTO validation via `class-validator` in NestJS.
- Rate limiting on auth endpoints, using Redis for store.
- JWT middleware ensures only session members can access `/sessions/:id/*` routes and join websockets.
- Turn validation ensures attacker/defender alignment with current turn and round counters.
- Mitigation text comparison uses fuzzy matching (Levenshtein) plus moderator cards list to allow synonyms. All correct answers stored in `cards.mitigations` for cross-locale support.

## Deployment Considerations

- Containerized services allow deployment on Render, Railway, Fly.io, or AWS ECS.
- Use managed PostgreSQL and Redis in production; configure secrets through environment variables.
- Horizontal scaling: multiple API instances coordinated through Redis Pub/Sub for Socket.io adapters.
- Observability: integrate NestJS with OpenTelemetry exporters and host logs via structured JSON.
